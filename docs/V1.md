## 2. V1 是什么

Synod = 自包含的多 agent 协作 CLI(纯 Node 20+,零三方依赖,内置后端直接 spawn 本地 coding agent)。V1 的目标:从 MVP 升级为正式项目——**三种一等使用面共享一个内核**:

1. **REPL 主持人模式**(保留并强化现有玩法):human 手动 `/open` 拉起平级 agent、`@label` 定向、`@all` 广播、`/relay` 转发;
2. **Workflow 模式**(第一阶段):人写 JS flow 的确定性编排,支持任意外部 CLI 接入、agent 档案配置、写隔离、断点恢复;
3. **Team 模式**(第二阶段):leader agent 自主拉起/协调 member,hub(leader 中转)与 direct(成员直连)两种交互模式。

前置是**进程治理**(阶段 0):任何退出路径零孤儿进程。

## 3. 决定登记簿(已拍板,不重新争论)

### 3.1 继承自 MVP 期(见 HANDOFF「不要重新争论的关键决定」)

## 5. 开发工法

### 5.1 每个 Task 的循环(TDD,计划已逐步写好)
写失败测试 → 跑确认失败(失败形态要与计划 Expected 一致)→ 最小实现 → 跑确认通过 → `npm test` 全量绿 → commit(用计划给的 message)。**禁止**:跳过 Red、一次做多个 Task、为过测试改测试(计划明示的契约变更除外)。

### 5.2 验证命令
```bash
npm test                  # Tier1 全量(零依赖,必须全绿才 commit)
npm run test:e2e          # A1–A8(需本机 omp/codex,缺则自动跳过)
npm run test:e2e-shutdown # 阶段 0 之后存在
node --test test/<file>   # 单文件快跑
```

### 5.3 「撰写计划」类条目的规则
按 writing-plans 规格(参考已出的两份计划的结构):标准头(Goal/Architecture/Tech Stack)→ Scope 与基线 → File Structure → bite-sized Task(每步含完整代码、确切命令、预期输出、commit)→ 完成定义 → 风险表 → Self-Review。**无占位符**(禁 TBD/“适当处理”);保存到 `docs/superpowers/plans/YYYY-MM-DD-架构重构-阶段X-<名>.md`;写完把本文 §4 对应「计划待撰写」勾掉并把链接补上。

### 5.4 执行主体(二选一,效果等同)
- **单 agent 直接开发**:本文 + plan 自足,逐 Task 做即可。
- **三方闭环**(本仓传统,见 HANDOFF「角色分工」):规划者把 plan 的 Task 原文当任务规格派给 deepseek(写代码+测试),codex 只读审+独立跑测,规划者 `git diff` + 亲跑 `npm test` 验收。派活运维坑见 HANDOFF「agent-bridge 运维要点」。

### 5.5 提交与分支
每阶段一条分支(看板已写名);Task 粒度 commit(message 在计划里);阶段 DoD 过了才合 main。不要求推送远端(用户决定)。
